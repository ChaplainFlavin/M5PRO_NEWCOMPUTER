# 纸箱厂轻量级智能订单流转系统 - 方案草案 v2.1

**最后更新**：2026-03-09（结合方案报告 2.0 复盘后修订）  
**项目背景**：传统纸箱厂设备老化、人员数字化能力有限，难以落地重型 ERP。目标是以即时通讯为入口，搭建低门槛、可验收、可持续运维的订单流转系统。

---

## 0. 边界与原则

### 0.1 一期边界
- 聚焦订单全流程：录入、排产、出货、回款。
- 不做复杂 MES 重构，不改造现有设备联网。
- 优先保证“可用、可追溯、可回滚”，再追求算法最优。

### 0.2 接入原则（强约束）
- **生产环境默认企业微信**（合规、API 稳定）。
- 个人微信仅用于 PoC，不作为长期生产主通道。

---

## 1. 核心设计理念

**把即时通讯工具变成轻量 OA，让 AI 在后台做结构化。**

```text
微信下单 -> AI录入 -> 定料采购 -> 排程规划 -> 投产追踪 -> 成品入库 -> 物流规划 -> 回款监控
```

---

## 2. 六大核心功能模块

### 模块一：智能订单录入
- 输入：文字、语音、手写表格照片。
- AI 处理：OCR + 字段提取 + 体积自动计算。
- 字段：客户、尺寸、材质、图案、数量、交期、单箱体积、总体积。
- 防误录：回显确认卡（确认/纠错后入库）。
- 低置信度处理：字段置信度低于阈值时，强制人工二次确认。

### 模块二：物料采购与到货追踪
- 自动计算原纸需求与库存缺口。
- 采购确认后记录 ETA，临近到货自动提醒。
- 到货确认后自动校验是否满足在途订单投产条件。

### 模块三：交付驱动排程（逆向规划）
- 先按送货区域聚合，再按交期倒推生产优先级。
- 优先级标签：`今日必做` / `明日安排` / `下周再做`。
- 支持“完工量变化后二次重排”，避免死计划。

### 模块四：生产工序极简追踪
- 工序：`开槽 -> 钉箱 -> 打包 -> 入库`。
- 工人群消息触发状态更新，记录时间戳和操作者。
- 看板实时展示卡点工序和延期风险。

### 模块五：物流装车与路线规划
- D-1 汇总待出货池，先交期后装载率优化。
- Bin Packing 计算装载组合，VRP 计算送货顺序。
- 老板确认后自动下发行程单给司机并通知客户。
- 失败降级：地图或算法异常时切换“区域分组 + 人工确认”兜底方案。

### 模块六：账期与回款监控
- 发货后自动挂账并生成最晚回款日。
- 每日逾期预警（客户、单号、金额、逾期天数）。
- 财务口令触发自动核销，累计客户回款画像。

## 3. 核心状态机与幂等规则（新增）

### 3.1 订单状态机

| 状态 | 进入条件 | 可流转到 |
|---|---|---|
| 草稿 | AI 初提取完成 | 待确认 / 已作废 |
| 待确认 | 销售待核对 | 待采购 / 已作废 |
| 待采购 | 缺料或待叫料 | 待投产 / 挂起 |
| 待投产 | 原料满足 | 生产中 / 挂起 |
| 生产中 | 工序已开始 | 待出货 / 挂起 |
| 待出货 | 打包入库完成 | 已发货 / 挂起 |
| 已发货 | 司机出车确认 | 已回款 / 部分回款 |
| 已回款 | 财务核销完成 | 结束 |

### 3.2 幂等与冲突处理
- 每条消息落库前生成 `msg_id + order_id + action` 幂等键，重复消息不重复执行。
- 同一订单 5 分钟内多次改价/改量触发“冲突待确认”。
- 任何状态回退必须带“原因 + 操作人 + 时间戳”审计记录。

## 4. 验收 KPI（新增，替代口号式目标）

| 维度 | 指标门槛（一期） | 统计口径 |
|---|---|---|
| 订单录入准确率 | >= 97%（关键字段） | 周报 |
| 漏单率 | <= 0.5% | 周报 |
| 工序更新时效 | P95 <= 1 分钟 | 日报 |
| 交付准时率 | >= 95% | 周报 |
| 装载率提升 | 较基线提升 >= 10% | 月报 |
| 逾期回款率 | 较基线下降 >= 20% | 月报 |
| 系统可用性 | >= 99% | 月报 |

## 5. 异常场景 SOP（新增）

1. 改单/撤单：自动冻结物流方案，重算排产优先级。
2. 部分完工：允许部分出货，剩余量保留原订单关联。
3. 司机临时变更：重发司机行程单，客户通知不变。
4. 地图 API 限流/失败：启用缓存路由矩阵，超时则切手工确认模式。
5. AI 提取失败：回退“人工录入模板”，不中断业务。
6. 服务异常：进入“只读看板 + 人工登记”应急模式，恢复后补录。

| 接入层 | 企业微信 | 生产首选 |

## 7. 成本评估（CAPEX + OPEX）

### 7.1 一次性成本（CAPEX）
- 流程梳理与字段标准化
- Bot 与表结构开发
- 基础看板与权限配置
- 培训与上线支持

### 7.2 月度成本（OPEX）
- 模型 API（按单量波动）
- 运维巡检与异常处理
- 数据备份与日志归档
- 接口变更适配

> 说明：`¥100-200/月`可作为低单量目标值，但应另行预留运维缓冲预算。

## 9. 数据安全与审计（新增）

- 角色分权：老板、销售、车间、财务、司机最小权限。
- 敏感字段脱敏：手机号、账款信息按角色可见。
- 每日自动备份与恢复演练（至少月度一次）。
- 关键动作审计：改单、撤单、核销、回退必须可追溯。

## 10. 关键待确认事项（保留并细化）

1. 最终接入是否确定为企业微信？
2. 每车体积与载重上限各是多少？是否有分车型？
3. 客户地址是否标准化（可地理编码）？
4. 峰值订单量、峰值在制单量分别是多少？
5. 库存是按订单隔离还是公共池？
6. 是否已有司机与客户联系人主数据？
7. 能否提供 20-50 份真实订单样本（文本+图片）用于阈值校准？
